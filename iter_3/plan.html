iter_3 Implementation Plan: Public Alternative Data + Rationale-Gated OOS CAGR

Goal: create iter_3/ as the next autonomous macro research repo, extending iter_2 from daily ETF adjusted closes into point-in-time public-domain alternative datasets—especially text—while requiring every tested idea to carry an explicit economic / academic / behavioral rationale and optimizing for out-of-sample CAGR.

Explicit product stance: program.md should state that this system is dispensing financial advice and that the research objective is to maximize out-of-sample CAGR. CAGR remains the target metric; DBMF excess CAGR is a required diagnostic, not the primary objective. Because of that, iter_3 must add stronger evidence, validation, audit trails, and risk notes than iter_2.

iter_3 should keep the successful simplicity of iter_2 but add a data layer, feature layer, idea registry, and OOS evaluation layer.

3. Rationale gate: every idea must explain why it should work

To reduce pure data mining, every experiment must include a rationale that is both machine-validated and human-viewable. Store the rationale directly in the main results.tsv row for the run, rather than splitting performance into one TSV and rationale into another. This keeps the human audit trail in a single spreadsheet-friendly file: every row shows both what was tested, why it was tested, and how it performed. The Python IdeaRationale object in train.py should still be validated before execution, but its fields should be exported into the same results.tsv row as the OOS metrics. The backtest should refuse to run if the idea lacks a rationale, evidence category, expected mechanism, OOS hypothesis, and falsification criteria.

Human-viewable unified ledger: results.tsv

Use TSV rather than JSON as the primary audit artifact because it is diffable, spreadsheet-friendly, and easy for the human user to scan. Keep long fields concise but complete; use semicolon-separated lists for datasets, assets, and references. results.tsv is append-only: each experiment gets one row containing both rationale fields and result fields.

Suggested idea schema

from dataclasses import dataclass
from typing import Literal

EvidenceType = Literal[
    "economic_intuition",
    "academic_literature",
    "behavioral_finance",
    "market_microstructure",
    "policy_transmission",
    "replication_of_known_anomaly",
]

@dataclass(frozen=True)
class IdeaRationale:
    idea_id: str
    title: str
    evidence_type: EvidenceType
    mechanism: str              # why the signal should forecast returns
    expected_assets: list[str]   # ETFs / asset classes affected
    feature_inputs: list[str]    # datasets/features used
    oos_hypothesis: str          # what should hold out of sample
    falsification: str           # what result would invalidate it
    references: list[str]        # papers, books, policy docs, URLs

    def validate(self) -> None:
        required = [self.idea_id, self.title, self.mechanism, self.oos_hypothesis, self.falsification]
        if any(not str(x).strip() for x in required):
            raise ValueError("IdeaRationale is incomplete")
        if len(self.references) == 0:
            raise ValueError("At least one reference or source note is required")

4. OOS-first evaluation design

iter_2 optimized one matched historical window. iter_3 should report in-sample diagnostics but decide keep/discard on OOS CAGR, with excess CAGR versus DBMF as a diagnostic guardrail.

Recommended default split

  • Warmup: earliest data through 2019, used only to compute initial lookbacks/features.
  • Development / train: 2020-01-01 through 2022-12-31.
  • Validation: 2023-01-01 through 2024-12-31.
  • Locked OOS: 2025-01-01 onward. This is the primary keep/discard criterion.

Also add walk-forward folds so a strategy cannot win by one lucky split.

@dataclass(frozen=True)
class EvaluationSplit:
    name: str
    start: str
    end: str | None
    role: Literal["warmup", "train", "validation", "oos"]

DEFAULT_SPLITS = [
    EvaluationSplit("warmup", "2018-01-01", "2019-12-31", "warmup"),
    EvaluationSplit("train", "2020-01-01", "2022-12-31", "train"),
    EvaluationSplit("validation", "2023-01-01", "2024-12-31", "validation"),
    EvaluationSplit("locked_oos", "2025-01-01", None, "oos"),
]

@dataclass(frozen=True)
class OOSResult:
    train_cagr: float
    validation_cagr: float
    oos_cagr: float
    benchmark_oos_cagr: float
    excess_oos_cagr_vs_dbmf: float
    ruined: bool
    passed: bool

Keep/discard rule

def passes_oos_gate(result: OOSResult, previous_best_oos_cagr: float) -> bool:
    if result.ruined:
        return False
    if result.oos_cagr <= previous_best_oos_cagr:
        return False
    if result.excess_oos_cagr_vs_dbmf <= 0.0:
        return False
    if result.validation_cagr <= 0.0:
        return False
    # Optional robustness guard: OOS CAGR should not be tiny relative to train CAGR.
    if result.train_cagr > 0 and result.oos_cagr < 0.25 * result.train_cagr:
        return False
    return True

Task 5: Add deterministic text features

HAWKISH = {"inflation", "tightening", "restrictive", "hike", "overheating"}
DOVISH = {"unemployment", "slowdown", "accommodative", "cut", "weakness"}
UNCERTAINTY = {"uncertain", "risk", "volatility", "fragile", "stress"}

def tokenize(text: str) -> list[str]:
    return [t.lower() for t in re.findall(r"[A-Za-z][A-Za-z-]+", text)]

def lexicon_score(text: str, positive: set[str], negative: set[str]) -> float:
    tokens = tokenize(text)
    if not tokens:
        return 0.0
    pos = sum(t in positive for t in tokens)
    neg = sum(t in negative for t in tokens)
    return (pos - neg) / max(len(tokens), 1)

def build_fomc_features(events: pd.DataFrame) -> pd.DataFrame:
    rows = []
    for _, row in events.iterrows():
        text = str(row["text"])
        rows.append({
            "published_at": row["published_at"],
            "hawkish_minus_dovish": lexicon_score(text, HAWKISH, DOVISH),
            "uncertainty_share": len([t for t in tokenize(text) if t in UNCERTAINTY]) / max(len(tokenize(text)), 1),
            "token_count": len(tokenize(text)),
        })
    return pd.DataFrame(rows).sort_values("published_at")

Task 6: Join features to trading dates without lookahead

All event features must be lagged to the next available trading day after publication. Monthly selectors should only see features published on or before the signal day.

def align_event_features_to_prices(
    prices: pd.DataFrame,
    events: pd.DataFrame,
    published_col: str = "published_at",
) -> pd.DataFrame:
    trading_days = pd.DataFrame({"date": prices.index})
    features = events.copy()
    features["published_at"] = pd.to_datetime(features[published_col], utc=True).dt.tz_localize(None)
    features["date"] = features["published_at"].dt.normalize()
    features = features.sort_values("date")
    aligned = pd.merge_asof(
        trading_days.sort_values("date"),
        features.sort_values("date"),
        on="date",
        direction="backward",
    ).set_index("date")
    return aligned.ffill()

7. Example text-aware strategy idea

This is a starter example, not the only idea. It combines price momentum with Fed-tone risk control.

def select_weights(day, prices, features, idea):
    idea.validate()

    # Base: current iter_2 winner, 21d top-1 thematic momentum.
    scores = []
    for symbol in SYMBOLS:
        if symbol == CASH_SYMBOL or symbol not in prices.columns:
            continue
        mom = lookback_return(prices[symbol], 21)
        if mom is not None:
            scores.append((mom, symbol))
    if not scores or max(scores)[0] <= 0:
        return {"SGOV": 1.0}

    latest = features.iloc[-1] if not features.empty else None
    hawkish = float(latest.get("hawkish_minus_dovish", 0.0)) if latest is not None else 0.0
    uncertainty = float(latest.get("uncertainty_share", 0.0)) if latest is not None else 0.0

    best_score, best_symbol = sorted(scores, reverse=True)[0]

    # Financial-advice rule under test: if Fed text is very hawkish/uncertain,
    # suppress long-duration/high-beta positions unless their momentum is very strong.
    high_beta = {"QQQ", "XLK", "SMH", "SOXX", "XBI", "XLY"}
    if best_symbol in high_beta and hawkish > 0.002 and uncertainty > 0.01 and best_score < 0.04:
        return {"SGOV": 1.0}

    return {best_symbol: 1.0}

8. Results ledger changes

Expand results.tsv so rationale, evidence, OOS hypothesis, falsification rule, and OOS performance are first-class fields in a single human-viewable ledger.

commit	timestamp_utc	idea_id	title	evidence_type	mechanism	expected_assets	feature_inputs	oos_hypothesis	falsification	references	human_notes	train_cagr	validation_cagr	oos_cagr	benchmark_oos_cagr	excess_oos_cagr_vs_dbmf	ruined	status	description

Why one TSV: keeping rationale and results together avoids split-brain audit trails. A human should be able to sort/filter results.tsv and immediately see whether a run was economically motivated, what data it used, what would falsify it, and whether it produced OOS CAGR and positive excess CAGR versus DBMF.

Status values:

  • keep: valid rationale, non-ruined, valid validation CAGR, positive OOS CAGR, positive OOS excess CAGR versus DBMF, and improves best OOS CAGR.
  • discard: ran successfully but failed the gate or did not improve best OOS CAGR.
  • invalid_rationale: missing or weak rationale; do not run or rank.
  • crash: code/data failure.

9. Analysis notebook requirements

analysis.ipynb should become an audit notebook, not just a performance plot. It should read the unified results.tsv and answer whether the research process is respecting the rationale gate before analyzing OOS CAGR.

Required notebook sections

  • Rationale coverage: count runs with missing idea_id, missing references, or blank mechanism/falsification fields in results.tsv.
  • Evidence mix: plot results by evidence_type such as economic intuition, academic literature, behavioral finance, and policy transmission.
  • Dataset mix: summarize which public datasets/features are being tested and whether text features are improving OOS CAGR.
  • Rationale vs performance: compare OOS CAGR distributions by rationale category; flag categories that look data-mined or consistently fail.
  • Gate failures: produce a human-readable table of rejected/invalid ideas and the reason they failed the rationale or OOS gate.
  • OOS frontier: plot the running best oos_cagr among valid-rationale ideas only, with invalid-rationale ideas shown separately.

Notebook code snippet

import pandas as pd

results = pd.read_csv("results.tsv", sep="\t")

required_cols = ["idea_id", "mechanism", "oos_hypothesis", "falsification", "references", "evidence_type"]
for col in required_cols:
    results[f"missing_{col}"] = results[col].isna() | (results[col].astype(str).str.strip() == "")

results["rationale_valid_for_analysis"] = ~results[
    [f"missing_{col}" for col in required_cols]
].any(axis=1)

print("Rationale gate coverage")
print(results["rationale_valid_for_analysis"].value_counts(dropna=False))

invalid = results[~results["rationale_valid_for_analysis"]][
    ["idea_id", "status", "description", *[f"missing_{c}" for c in required_cols]]
]
display(invalid)

valid = results[results["rationale_valid_for_analysis"]].copy()
summary = valid.groupby("evidence_type").agg(
    ideas=("idea_id", "count"),
    median_oos_cagr=("oos_cagr", "median"),
    best_oos_cagr=("oos_cagr", "max"),
    keep_rate=("status", lambda s: (s == "keep").mean()),
).sort_values("best_oos_cagr", ascending=False)
display(summary)

13. Acceptance criteria

  • iter_3 exists and is self-contained.
  • program.md explicitly says the system is dispensing financial advice and aims to maximize OOS CAGR.
  • At least one public text dataset is ingested into a normalized, timestamped event table.
  • At least one deterministic text feature is generated and point-in-time aligned to trading dates.
  • Backtests include DBMF matched benchmark plus train/validation/OOS CAGR metrics and excess CAGR diagnostics.
  • Every run has a validated IdeaRationale; missing rationale blocks ranking.
  • results.tsv is readable by a human as the unified performance+rationale ledger, with one row per experiment and enough context to judge whether the idea is economically motivated rather than pure data mining.
  • Tests prove no lookahead in price or text feature slicing.
  • The first iter_3 baseline reproduces the iter_2 price-only strategy before adding text features, so improvements can be attributed.
Recommended first milestone: build only FOMC text + rationale schema + OOS split first. Once that is stable, add Beige Book and SEC filings. This avoids turning iter_3 into a broad data swamp before the evaluation discipline is in place.
